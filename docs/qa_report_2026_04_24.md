## Open issues at end of session (still to verify on real devices)

### BUG-A — Inbound envelope arrives but UI never updates

**Symptom (2026-04-24, 19:26 emulator log):**

```
PhantomRelay I Received envelope: id=4554625b-372 sealed=true payloadBytes=1368
```

But the chat row never appears in ChatListScreen and no notification fires.

**Diagnostics added (already in this branch):**

`shared/core/messaging/src/commonMain/.../DefaultMessagingService.kt`
now logs every step of `handleDeliver` with tag `PhantomMessaging`:

```
handleDeliver start: id=…
Sender identified: …
Session loaded: conv=…
Decrypt OK: plaintextBytes=…
Payload parsed: type=… textLen=…
Inserting message into DB: id=…
DB insertMessage OK
Emitting IncomingMessage to UI flow
Invoking onNewMessageNotification callback (null=false)
handleDeliver DONE for id=…
```

The previous `runCatching { ... }.onFailure { _ -> }` block (which silently
swallowed every error in this pipeline) has been replaced with an
`.onFailure { e -> messagingLog(ERROR, ..., e) }` so any decrypt /
storage / callback throw now lands in logcat with full stack.

**Leading hypothesis (code-only analysis):**

The first message from a sender who has *not* been added to the
recipient's contacts goes to **Message Requests**, not to the main chat
list, by design. The exact path:

1. `handleDeliver` calls `conversationRepository.getConversation(convId)`.
2. If `existing == null`, the new conversation is upserted with
   `trustTier = TrustTier.REQUEST`
   ([`DefaultMessagingService.kt:328`](../shared/core/messaging/src/commonMain/kotlin/phantom/core/messaging/DefaultMessagingService.kt)).
3. `getActiveConversations` SQL query filters
   `WHERE blocked = 0 AND trust_tier = 'TRUSTED' AND … AND archived = 0`
   ([`Conversation.sq:21-22`](../shared/core/storage/src/commonMain/sqldelight/phantom/core/storage/Conversation.sq)),
   so a `REQUEST` conversation does **not** appear in `ChatListScreen`.

**Why both directions look broken in QA:**

QA flow was: User A pasted User B's `username:key`, then sent a
message. User B may not have pasted User A's key first. So on User B
the conversation is unknown → `REQUEST` → invisible on the main chat
list. The message *is* delivered and stored — it just lives behind
"Message Requests" until accepted.

**Confirm tomorrow with logs:**

```bash
adb logcat -c
adb logcat PhantomRelay:V PhantomMessaging:V PhantomUI:V *:S
```

After both devices receive a message, look for:

- `handleDeliver DONE` — the inbound pipeline completed cleanly.
- `getActiveConversations` does not surface the new conversation,
  but `getMessageRequests` does → `requestCount` chip updates on
  ChatListScreen.

**Possible follow-up fixes (not done yet — needs UX decision):**

Option A. When sender's `theirPublicKeyHex` is already in the local
identity store as a known peer (e.g. recipient previously added them),
treat the first inbound as `TRUSTED`. Today this only happens if the
recipient explicitly added the sender via `AddContactDialog`.

Option B. Add a clearer "1 new message request" banner above the
chat list when `requestCount > 0`, so testers don't mistake it for
"nothing happened."

Option C. Auto-promote a `REQUEST` to `TRUSTED` after the recipient
sends *any* reply, and leave the existing manual accept for cases
where they don't reply. Today `acceptRequest` is the only path.

Recommendation: Option B for Alpha-1 (5-line UI change), Option A
deferred to Alpha-2 once the proper directory lookup lands.

### BUG-B — `send() skipped — not connected. state=Connecting`

**Symptom:** envelopes silently dropped when the user hits Send during
a reconnect window. Caddy idle timeout closes quiet sockets every few
minutes; if Send fires inside the reconnect window the message is lost
with only a WARN log.

**Status:** ✅ **FIXED** in commit `216aa1db`.

`KtorRelayTransport` now keeps an in-memory FIFO `pendingOutbox`. When
not connected, `send()` enqueues and returns false (so
`MessageRepository` keeps the row at `MessageStatus.QUEUED`). The next
`Connected` transition triggers `flushPendingOutbox()`. The same applies
to read receipts. Typing events stay ephemeral.

The `RECONNECT_MAX_ATTEMPTS = 5 — giving up` failure mode is also gone.
The reconnect loop is now infinite with exponential backoff capped at
30 s. Pong-timeout-driven reconnect closes the socket if the relay
falls silent for 50 s.

**Verify tomorrow:**

1. Disable Wi-Fi briefly, hit Send, re-enable. Expected:
   `PhantomRelay I Queued until reconnect: id=…`
   `PhantomRelay I WebSocket connected successfully`
   `PhantomRelay I Flushing 1 queued item(s) after reconnect`
   `PhantomRelay I Flush → send envelope: id=…`
2. ChatListScreen now shows a thin status banner: `Offline — messages
   queued` (red dot). On reconnect within 1.5 s the banner does not
   appear at all (grace timer); longer outages surface it.

### BUG-D — Phone crash on `Scan QR Code` (`jb2.run` NPE)

**Symptom:** tapping `Scan QR Code` on the phone produced a FATAL
EXCEPTION with `NullPointerException` at `jb2.run` /
`Handler.handleCallback`.

`jb2` did not appear in the QA-time mapping.txt because the crashing
APK was a previous build. The current mapping resolves
`CameraPreview$lambda$26$lambda$25$lambda$24$lambda$23` to the
ImageAnalyzer frame handler — the most likely culprit even when
the symbol stops being `jb2`.

**Status:** ✅ **FIXED** in commit `3d4d9577`.

`apps/android/src/androidMain/.../qr/QrScanScreen.kt` rewritten:

- Outer `addListener { ... }` wrapped in try/catch — exceptions during
  `cameraProviderFuture.get()`, `bindToLifecycle()`, etc., now log
  with tag `PhantomQR` instead of crashing.
- Analyzer frame handler also try/catch'd; `imageProxy.close()` runs
  on every path (success, failure, malformed image).
- `scanner.process(image)` now has explicit
  `addOnSuccessListener` + `addOnFailureListener` + `addOnCompleteListener`.
- `DisposableEffect.onDispose` shuts down the analyzer executor and
  unbinds the camera provider — late callbacks on a destroyed
  composable can no longer NPE.

`apps/android/proguard-rules.pro` extended with
`-keep class androidx.camera.**`,
`-keep class com.google.mlkit.**`, and
`-keep class com.google.android.gms.vision.**`. Without these R8
obfuscated the ML Kit reflective entry points and the symbolic stack
trace was unreadable even to the developer.

**Verify tomorrow:**
Tap `Scan QR Code` → camera preview loads → scan a known QR. Filter
logcat with `PhantomQR:V`:

```
PhantomQR I QR camera preview bound to lifecycle
(... scan happens ...)
PhantomQR I Disposing QR preview — unbinding camera + shutting down executor
```

If a crash recurs, the new logging pinpoints which step (provider
init, lifecycle bind, analyzer frame, ML Kit success handler) and
the keep rules give a readable stack trace.

### BUG-E — input field runs off the screen on the phone

**Symptom (verbal report at session end):** when the soft keyboard
appears in `ChatScreen`, the message input bar slides under the
keyboard.

**Status:** ✅ **FIXED** in commit `216aa1db`.

`apps/android/src/androidMain/.../screens/chat/ChatScreen.kt`:
`Modifier.imePadding()` added on the root Scaffold. On API ≥ 35 the
system handles edge-to-edge automatically. On API < 35 (the Tecno phone
runs API 31), `WindowCompat.setDecorFitsSystemWindows(window, false)`
in `MainActivity.onCreate()` routes IME insets through Compose; without
`imePadding()` on Scaffold the bottomBar (which contains the input
field) was positioned below the keyboard.

The same pattern was already applied to the Onboarding screen earlier
in this branch (commit `e618b82f`).

## Reproduction recipe — first E2E message on tomorrow's session

1. Cold boot the emulator (Android Studio → Device Manager → Cold Boot
   Now). Confirm `adb -e shell ping -c 2 relay.phntm.pro` resolves.
2. Reinstall on both devices:
   ```bash
   adb -s <id> uninstall phantom.android
   adb -s <id> install apps/android/build/outputs/apk/release/android-release.apk
   ```
3. Confirm signature parity:
   ```bash
   ~/AppData/Local/Android/Sdk/build-tools/37.0.0/apksigner.bat verify \
       --print-certs apps/android/build/outputs/apk/release/android-release.apk \
       | grep SHA-256
   # expect: aa1709483ebd47f121ce0bd14692d1d575fd28a0d65c4b2e201abe882cab1f02
   ```
4. Onboard each device with a different username.
5. Profile → Share my key on each, copy `username:key`.
6. **Both directions:** paste the *other* device's `username:key` into
   the Add Contact dialog **before** sending any message. This is the
   step that prevents the conversation from going to Message Requests.
7. Open the chat from User A side, send "hi". Expected on User B:
   - `PhantomRelay I Received envelope: id=… sealed=true`
   - `PhantomMessaging I handleDeliver start: id=…`
   - `PhantomMessaging I … Decrypt OK …`
   - `PhantomMessaging I DB insertMessage OK`
   - `PhantomMessaging I Emitting IncomingMessage to UI flow`
   - `PhantomMessaging I handleDeliver DONE for id=…`
   - Notification appears + chat row updates in `ChatListScreen`.

8. If the message *still* does not appear in the main chat list but
   `handleDeliver DONE` is logged, run:
   ```bash
   adb -s <id> shell run-as phantom.android sqlite3 \
       /data/data/phantom.android/databases/phantom.db \
       'SELECT id, their_username, trust_tier FROM conversation;'
   ```
   If `trust_tier = 'REQUEST'`, BUG-A hypothesis (Option B fix needed)
   is confirmed.
